# CLAUDE.md

Personal collection of single-page HTML cheatsheets for daily-driver tools (neovim, tmux, omarchy, …). Each sheet has a markdown source of truth and a generated, styled HTML output.

**Core rule:** never edit content directly in `docs/*.html`. Edit the markdown source in `src/`, then regenerate the HTML. The exception is visual/CSS refinement work, which happens in the HTML — when a refinement should become the standard for all sheets, back-port it into this file and the reference implementation.

## Repository layout

```
templates/base.html    canonical template — skeleton + all shared CSS (the theme)
src/<tool>/            one folder per tool, holding all source material:
src/<tool>/<tool>.md     the canonical content file — the source of truth for content
src/<tool>/*             supplementary material: screenshots of other cheatsheets,
                         PDFs, scratch notes, exported configs, …
docs/<tool>.html       generated, self-contained HTML cheatsheet
```

Supplementary files are **input material, not sources of the HTML**: when asked to incorporate them (e.g. "add the bindings from this screenshot"), read/extract their content into `src/<tool>/<tool>.md` first — the HTML is always generated from the canonical markdown alone. Keep the material in the folder afterwards for provenance.

`docs/` is used (rather than `dist/` or `out/`) so the sheets can be hosted with GitHub Pages later without restructuring.

## Markdown source format

- **H1** = sheet title. A **blockquote** directly after it = subtitle (inline code allowed).
- **H2** = one card (topic group). An optional trailing parenthesized hint becomes the card head's small note: `## Normal (movement & editing)`.
- Under each H2, a table with columns `| Keys | Action |`.
  - Each inline code span in the Keys cell renders as one key-cap: `` `w` `b` `` → two key-caps; `` `h j k l` `` → a single key-cap containing the sequence.
  - A trailing `(custom)` in the Action cell marks a personal (non-default) binding; render it with a visible marker (badge or distinct row style).
- Layout is hand-tuned per sheet and expressed as HTML comments, e.g. `<!-- layout: wide -->` after an H2, or a sheet-level `<!-- layout: ... -->` block describing grid placement. Same for other presentation metadata: `<!-- statusbar: ... -->`, `<!-- footer: ... -->`, `<!-- flourish: ... -->`.

See `src/neovim/neovim.md` for a complete example.

## HTML design system

`templates/base.html` is the **canonical source of all shared styling and structure**. Sheets are generated by filling in its `{{…}}` placeholders and appending sheet-specific CSS below the marked `SHEET-SPECIFIC STYLES` divider (accent mapping, `.pos-*` grid placement, flourishes). `docs/neovim.html` is a full worked example.

- **Theme:** Tokyo Night palette defined as CSS variables (`--bg`, `--bg-dark`, `--bg-soft`, `--panel`, `--fg`, `--fg-dim`, `--rule`, and accents `--red` `--green` `--yellow` `--blue` `--purple` `--aqua` `--orange`). Inter for prose, JetBrains Mono for keys and chrome.
- **Anatomy, top to bottom:** `.statusbar` (segmented bar: active segment, tool name, path, right-aligned sheet title) → `.titlerow` (`h1` + `.subtitle`) → responsive card grid (`.modes`) of `.mode-card`s → `footer` one-liner.
- **Cards:** each `.mode-card` has a colored `.mode-head` (one accent color from the palette per card, dark text) with an optional small `.count` note, then `.rows` of `.row` = `.keys` (one or more `.key` key-caps) + `.desc`. Wide cards (`.wide`) span the full grid width for large groups. `(custom)` bindings get a `<span class="custom">custom</span>` badge in the `.desc`.
- **Self-contained:** one HTML file per sheet, all CSS inline, no JavaScript. The Google Fonts `@import` is the only permitted external dependency; font stacks must include system fallbacks so sheets degrade gracefully offline.
- **Responsive:** breakpoints at 1200px and 640px, collapsing the grid to 2 and 1 columns as in the reference.
- **Flourishes:** tool-specific decorative elements (like the hjkl cross on the neovim sheet) are encouraged but optional; they belong to that sheet only, not the shared system.

## Regeneration workflow

When asked to update a sheet:

1. If new source material is involved (screenshots, PDFs, notes in `src/<tool>/`), extract its content into `src/<tool>/<tool>.md` first.
2. Read `src/<tool>/<tool>.md` (apply the requested content change there first if it's a content change).
3. Regenerate `docs/<tool>.html` in full from `templates/base.html` and the source's layout comments.
4. Preserve any sheet-specific flourishes and CSS already present in the existing HTML (everything below the `SHEET-SPECIFIC STYLES` divider).

When asked to create a new sheet, create the folder and canonical markdown first, then the HTML.

**Theme/style changes** (new palette, typography, card styling, …) go the other direction: edit `templates/base.html` first, then regenerate **every** `docs/*.html` from its `src/*/[tool].md` so all sheets pick up the change. If a style refinement was prototyped directly in one sheet's HTML, back-port it into the template before regenerating the rest.

## Content guidelines

- Tool **defaults are the baseline**; personal mappings are included but marked `(custom)`.
- Action descriptions: concise, lowercase, no trailing period (`delete line`, not `Deletes the line.`).
- Keep each sheet to roughly one screen/printed page; prefer trimming rare bindings over shrinking text.
